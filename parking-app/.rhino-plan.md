# Implementation Plan: Parking App

## Summary
A multi-tenant parking management application built on Rhino V4 (Rails API + React frontend). Organizations manage parking lots, spots, vehicles, and reservations.

## PRD gaps compensated
None — PRD sections are aligned.

## Blueprint inventory
A blueprint YAML has been written for each of these models. Develop stage will run `bundle exec rails rhino:blueprint` to materialize them.

- `.agentcode/blueprints/_roles.yaml` — roles: admin, manager, member
- `.agentcode/blueprints/parking_lots.yaml` — ParkingLot
- `.agentcode/blueprints/parking_spots.yaml` — ParkingSpot
- `.agentcode/blueprints/vehicles.yaml` — Vehicle
- `.agentcode/blueprints/reservations.yaml` — Reservation

## Models (reference)

### ParkingLot
- Ownership: organization-scoped
- Key fields: `name` (string, required), `address` (string, required), `total_spots` (integer, required)
- Relationships: belongs to `Organization`, has many `ParkingSpot`

### ParkingSpot
- Ownership: nested under `ParkingLot` (cascades to organization)
- Key fields: `number` (string, required), `spot_type` (string, default: "standard"), `is_available` (boolean, default: true)
- Relationships: belongs to `ParkingLot`, has many `Reservation`

### Vehicle
- Ownership: organization-scoped
- Key fields: `license_plate` (string, required, unique per org), `make` (string), `model` (string), `color` (string), `vehicle_type` (string, default: "car")
- Relationships: belongs to `Organization`, belongs to `User`, has many `Reservation`

### Reservation
- Ownership: nested under `Vehicle` and `ParkingSpot` (cascades to organization)
- Key fields: `start_time` (datetime, required), `end_time` (datetime, required), `status` (string, default: "pending"), `total_cost` (decimal), `notes` (text)
- Relationships: belongs to `Vehicle`, belongs to `ParkingSpot`, belongs to `User`

## Extras (NOT covered by blueprint — develop stage implements these by hand)

### Vehicle ownership check (Vehicle#update, Vehicle#destroy)
- **Rule:** Members can only update or destroy their own vehicles.
- **Where the code lives:** `app/policies/vehicle_policy.rb`
- **Acceptance criterion:** AC-3

### Reservation ownership check (Reservation#store, Reservation#show)
- **Rule:** Members can only view or create reservations for their own vehicles / user account.
- **Where the code lives:** `app/policies/reservation_policy.rb`
- **Acceptance criterion:** AC-4

### ParkingSpot availability update (Reservation lifecycle)
- **Rule:** ParkingSpot `is_available` updates when a Reservation goes active/completed/cancelled.
- **Where the code lives:** `app/models/reservation.rb` (callbacks or state transition hooks)
- **Acceptance criterion:** AC-2

### Vehicle license plate uniqueness per organization
- **Rule:** Vehicle license plate must be unique within the scope of the organization.
- **Where the code lives:** `app/models/vehicle.rb`
- **Acceptance criterion:** AC-3

## Implementation order (recommended stage flow)

1. **Plan** — this artifact set (YAMLs, plan, criteria, index).
2. **Develop** — phase A: `bundle exec rails rhino:blueprint` + migrations. Phase B: implement **Extras** (callbacks, custom policies, validations). Phase D: write backend unit + integration tests for every AC.
3. **UI / UI-Integration** — verify auto-generated CRUD screens.
4. **Test** — run `bundle exec rails test` or `bundle exec rspec` to confirm green.
5. **Review** — PRD compliance + test/coverage report.

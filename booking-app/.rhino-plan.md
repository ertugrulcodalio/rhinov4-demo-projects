# Implementation Plan: BookEase

## Summary
BookEase is a hybrid booking platform built on the Rhino V4 Rails API framework. It allows service businesses to manage services, staff, and availability through an organization-scoped multi-tenant API, while customers browse active services and available time slots and create bookings through a single-tenant public route.

## PRD gaps compensated
- **KEY_FEATURE_NO_SITEMAP**: The PRD lacks a formal sitemap section. We have defined the screens and routes in this plan based on the route groups and permissions matrix.
- **DATA_MODEL_UX_MISMATCH**: The `time_slots` model table in Section 6 does not list `organization_id`, but Section 4 states it is scoped by `organization_id`. We have explicitly added `organization_id` to the `time_slots` blueprint and migration to ensure robust multi-tenant scoping and indexing.

## Models (reference)

### Service
- Ownership: organization-scoped (`belongs_to_organization: true`)
- Key fields: `name` (string, non-nullable), `description` (text), `duration_minutes` (integer), `price` (decimal), `status` (string, default: "draft")
- Relationships: `belongs_to :organization`, `has_many :time_slots`

### StaffMember
- Ownership: organization-scoped (`belongs_to_organization: true`)
- Key fields: `name` (string, non-nullable), `email` (string), `role_title` (string)
- Relationships: `belongs_to :organization`, `has_many :time_slots`

### TimeSlot
- Ownership: organization-scoped (`belongs_to_organization: true`)
- Key fields: `service_id` (foreignId, non-nullable), `staff_member_id` (foreignId, nullable), `starts_at` (datetime, non-nullable), `ends_at` (datetime, non-nullable), `available` (boolean, default: true)
- Relationships: `belongs_to :organization`, `belongs_to :service`, `belongs_to :staff_member`, `has_one :booking`

### Booking
- Ownership: organization-scoped (`belongs_to_organization: true`)
- Key fields: `user_id` (foreignId, non-nullable), `time_slot_id` (foreignId, non-nullable), `status` (string, default: "pending"), `notes` (text)
- Relationships: `belongs_to :organization`, `belongs_to :user`, `belongs_to :time_slot`

## Extras (NOT covered by blueprint — develop stage implements these by hand)

### ServiceScope
- **Model + action:** `Service#index`
- **Rule:** On the customer route, filter services to `status: active`. Staff/admins see all services.
- **Where the code lives:** `app/models/scopes/service_scope.rb`
- **Acceptance criterion:** AC-9, AC-10

### TimeSlotScope
- **Model + action:** `TimeSlot#index`
- **Rule:** On the customer route, filter time slots to `available: true`. Staff/admins see all slots.
- **Where the code lives:** `app/models/scopes/time_slot_scope.rb`
- **Acceptance criterion:** AC-9, AC-10

### BookingScope
- **Model + action:** `Booking#index`
- **Rule:** On the customer route, filter bookings to `user_id: current_user.id`. Staff/admins see all bookings in their organization.
- **Where the code lives:** `app/models/scopes/booking_scope.rb`
- **Acceptance criterion:** AC-8

### Booking Availability Callback
- **Model + action:** `Booking#create`
- **Rule:** Creating a booking marks the associated `TimeSlot` as unavailable (`available: false`).
- **Where the code lives:** `app/models/booking.rb` (after_create callback)
- **Acceptance criterion:** AC-5

### Booking Cancellation Policy
- **Model + action:** `Booking#update`
- **Rule:** Customers can only update their own bookings to set status to `cancelled` if the current status is `pending`.
- **Where the code lives:** `app/policies/booking_policy.rb`
- **Acceptance criterion:** AC-6

## Screens
Since there is no UI prototype, Rhino's auto-generated CRUD screens will be used for all resources.
- **Staff Service Management:** `/api/:organization/services` (auto-generated CRUD)
- **Staff Availability Management:** `/api/:organization/time_slots` (auto-generated CRUD)
- **Staff Booking Management:** `/api/:organization/bookings` (auto-generated CRUD)
- **Customer Catalog:** `/api/customer/services` and `/api/customer/time_slots` (auto-generated CRUD with custom scopes)
- **Customer Bookings:** `/api/customer/bookings` (auto-generated CRUD with custom scopes)

## Implementation order (recommended stage flow)

1. **Plan** — This planning stage (blueprints, plan, criteria, index).
2. **Develop** — Run `bundle exec rails agentcode:blueprint` to generate models and migrations. Implement custom scopes, callbacks, policies, and seeds. Write unit and integration tests.
3. **Test** — Run `bundle exec rails test` to verify all isolation and business logic tests pass.
4. **Review** — Verify PRD compliance and test coverage.
